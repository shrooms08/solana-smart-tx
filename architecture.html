System Architecture Document

Smart Transaction Stack
getting a bundle into a block, on time, and knowing for certain whether it landed.

A Rust service that submits Solana transactions through Jito bundles with the discipline a production system needs. It watches the leader schedule over a streaming connection, prices its tip from live auction data, tracks every submission through its full commitment lifecycle, classifies failures from real evidence, and lets an AI agent own one operational decision: what to do when a bundle does not land.

8 library crates + binary Yellowstone gRPC Jito Block Engine + BAM SQLite lifecycle AI failure reasoning

Central finding

Driven to where every operator-controlled variable is provably correct, a contentless bundle (memo plus self-transfer) still does not win a Jito auction. Confirmed with Jito support: a returned bundle id means received, not selected. The auction rewards economic value, not a bundle's mere existence.

00

What the system does

The job is narrow and unforgiving. On Solana, a transaction that is correct but late misses its slot. A bundle that is well-formed but under-tipped loses its auction. A confirmation read from a single RPC poll can lie by omission. The system is designed around these failure modes rather than the happy path, because the happy path is the part that takes care of itself.

On every run it:

  1. Streams slot updates and transaction statuses from a Yellowstone gRPC endpoint, so it always knows the current slot and can see its own transactions land.
  2. Computes which upcoming slots belong to Jito-enabled leaders, and fires a submission window one to two slots ahead of such a leader, so the bundle is buffered at the block engine before that leader begins producing.
  3. Prices the tip from the live Jito tip-floor stream rather than a hardcoded constant.
  4. Builds a bundle and submits it through an authenticated, rate-limited path to the block engine.
  5. Tracks the submission through Submitted, Processed, Confirmed, Finalized, stamping the slot and timestamp at each transition.
  6. When a bundle does not reach a terminal state, classifies why from the evidence available, and hands that classification to an AI agent that decides whether to retry, reprice, refresh the blockhash, or give up.

02

The crates

Eight library crates plus a binary. Each has one responsibility and a small surface area, so a problem can be localised to a single crate and tested there. For each, the responsibility and the one design decision worth knowing.

stream

Yellowstone gRPC ingestion: slot updates (the clock) and transaction statuses, under a reconnect supervisor that reconciles across gaps.

Asymmetric backpressureA dropped slot is harmless; the next arrives in ~400ms. A dropped tx status can mean missing a confirmation. Slots use a droppable try_send (freshness wins); statuses use an awaiting send that applies backpressure (delivery guaranteed). One stream, two delivery contracts.

leader

Intersects the RPC leader schedule with the Jito validator set, matching on the validator's identity account, and reports when the next Jito-led slot is near.

Submit ahead, by leader groupLeaders run in groups of four slots. Since Jito holds most stake, the current slot is almost always already Jito-led, tempting a late submission. Instead the crate targets the start of the next group strictly ahead (current minus current mod 4, plus 4) and fires only one to two slots out.

tips

Converts Jito's live tip-floor websocket stream (REST fallback) into lamports, reporting p50, p75, p95.

A feed, not a policyIt reports what the market is doing and leaves the choice of percentile to the orchestrator. No tip is ever hardcoded; every tip is derived from live data, and the percentile is configurable.

submitter

Builds the two-transaction bundle (memo plus self-transfer, then the tip transfer last) and submits it to the block engine.

One network call in the hot pathThe blockhash cache refreshes every two seconds and tip accounts are pre-warmed, so at submit time the only network call is sendBundle itself. Every block-engine call routes through a single rate-limited choke point that attaches the x-jito-auth header, with sendBundle prioritised over background traffic.

lifecycle

Persists submissions to SQLite and advances them through Submitted, Processed, Confirmed, Finalized, stamping slot and timestamp at each transition.

Stream-driven, not RPC-polledA single RPC poll sees only the moment it is asked; a bundle can land between polls. The tx-status stream delivers every transition as it happens. RPC polling is used only to reconcile after a gap. A timeout sweeper moves never-landed bundles to a terminal state.

failure

A pure classifier: given the evidence about a failed submission, returns a failure kind with a confidence level and rationale. No state, no network, exhaustively testable.

Weigh competing causesA never-landed bundle whose blockhash has since aged looks like an expiry, but if it lost its auction the expiry is a downstream symptom. Transport errors (including the rate-limit response) are checked first; ambiguity is named, not hidden behind false confidence.

agent

Owns one real operational decision: given a classified failure, retry as-is, raise the tip, refresh the blockhash and retry, or abort.

Shadow baseline + guardrailEvery decision runs alongside a deterministic baseline in an A/B shadow, and both are persisted. Proposed actions pass through a clamp that bounds them. If the model call fails, the system falls back to the baseline rather than stalling.

orchestrator

The binary. Composes the crates into a run loop: warm up, wait for a window, price, submit, track, and on failure invoke the agent and act.

Drain to terminalIt drains every submission to a terminal state before exiting, so a run produces a complete, honest record rather than leaving bundles in flight. Modes cover normal runs, fault injection, log export, and status inspection.

runtime

A small leaf crate shared by the others: one-time crypto-provider init, and credential redaction helpers for URLs and secrets.

Redaction by defaultURLs and tokens pass through masking helpers before they reach a log line, so endpoints and keys never leak into output.

03

Submission lifecycle

One normal submission, end to end:

  1. Warmup. stream connects and delivers slots; leader fetches the schedule and Jito set; tips connects; the blockhash and tip-account caches fill. The orchestrator waits until all three signals are live.
  2. Window. leader reports that a Jito leader's slot group begins one to two slots ahead, and whether that leader is BAM.
  3. Price. The orchestrator reads the configured percentile and sets the tip. On a BAM leader with BAM pricing enabled, it also sets a priority fee and a compute-unit limit.
  4. Build and send. submitter reads the cached blockhash and a tip account from memory, builds and signs the bundle, and sends it through the single authenticated, rate-limited choke point.
  5. Track. The returned bundle id and memo signature are recorded; lifecycle begins watching the tx-status stream for the memo signature.
  6. Resolve. Either the memo signature appears and lifecycle advances the bundle through its commitment states, or the timeout sweeper moves it to a terminal never-landed state.
  7. On failure. failure classifies the cause from evidence; agent decides the response; the orchestrator acts, bounded by a maximum attempt count.

Commitment transitions are instrumented end to end. In the recorded runs no bundle reached Processed, so no live commitment-latency deltas exist to report; the values discussed below are derived from Solana's slot timing and labelled as such.

04

The latency investigation

The most instructive part of building this system was discovering why, for a long time, no bundle would land. The method throughout: when a bundle did not land, do not theorise, measure something that distinguishes the candidate causes. Capture the raw response. Decode the bytes on the wire. Simulate the exact signed transaction. Query the source of truth directly. Reasoning from absence repeatedly misled; reasoning from a captured measurement repeatedly resolved.

The hot path, before and after

Timing telemetry showed the in-window work was dominated by RPC round-trips and, hidden inside the send, a full TLS handshake re-established on every call. Pre-fetching the blockhash and tip accounts and pooling one warm connection collapsed it:

05

The two auctions

A Jito-enabled validator may run one or two auctions, and they score bundles differently. The system surfaces an is_bam bit per leader so it can price for whichever auction the target runs.

Block Engine auction

every Jito validator

priority ≈ tips / CU

Only the tip matters, and a tight compute footprint helps. A contentless bundle competes on an axis where it has no structural advantage against real MEV flow.

BAM Node auction

BAM validators only (387 of 688 in one observed run)

priority ≈ (tips + priority fees) / CU

Priority fees count too, so a tip-only bundle competes with half the formula missing. The submitter can attach a priority fee paired with an explicit compute-unit limit.

Why the compute-unit limit is paired The fee is price × requested-CU-limit. Without an explicit limit the default request inflates the denominator several-fold, both making the fee unpredictable and collapsing the (tips + fees) / CU ratio the auction actually scores. Pinning the limit keeps the denominator tight and the fee deliberate.
06

Failure classification & the AI decision

The system is built around failure, not the happy path. It produces and classifies real failures, then lets the agent decide the response.

  • TransportError — network and rate-limit failures, including the block engine's HTTP 429, checked first so they are never misread as a pricing or construction fault.
  • FeeTooLow — the tip was below what the auction required.
  • ExpiredBlockhash — the referenced blockhash was genuinely stale at submission, not merely aged while waiting.
  • AuctionLost — the block engine accepted the bundle (a bundle id was returned) but the status API reported Invalid: it did not win its auction. The aged blockhash on such a bundle is named a downstream symptom, not the cause.

The corrected classification matters

An earlier version labelled every never-landed bundle ExpiredBlockhash, because the blockhash had aged past 150 slots by the time the sweeper ran. But that age accrued while the bundle sat unlanded after losing its auction. The fix introduced AuctionLost, plumbed the persisted Jito verdict into the evidence the classifier sees, and reserved ExpiredBlockhash for blockhashes stale at submission. The result reads honestly: the failures are auction losses, which is both the true story and the more sophisticated one.

Across those decisions the agent visibly tracked a live market: raising the tip into rising auctions, declining to overpay into cooling ones, and separating a competitive-tip-but-stale-blockhash from a genuine needs-higher-bid. The Certain versus Ambiguous split reflects whether the poller captured a definitive Invalid before the sweep. And at one point the LLM call failed and the deterministic baseline took over with a refresh-and-resubmit: the resilience path, demonstrated in the log rather than merely claimed.

Honest scope: leader identity and the is_bam flag are not persisted per row, so they are omitted per bundle rather than inferred. No bundle reached Processed, Confirmed, or Finalized, so there are no commitment-latency deltas to report.

07

Findings

The central finding came from driving the system to the point where every variable under the operator's control is provably correct, and then asking what remained.

With construction, execution, funding, tip, timing, authentication, region, and BAM priority fees all eliminated by measurement, the block engine consistently accepts the bundle and then reports it Invalid: it did not win its auction. The conclusion the evidence supports is that a bundle carrying no economic activity does not win a Jito auction against the real arbitrage and swap flow it competes with, and that no amount of tip, priority fee, authentication, timing precision, or region selection changes that, because the auction rewards the value a bundle brings to the block, not the bundle's presence.

What landing would require Give the bundle genuine economic content: a real on-chain action whose profit funds the tip. At that point it competes on the same axis as the flow that currently beats it. The infrastructure built here, streaming, leader targeting, live tip pricing, low-latency authenticated submission, lifecycle tracking, classification, and the agent layer, is exactly what such a strategy sits on top of. The hard part of a smart transaction stack is the machinery, and the machinery is what this project built and verified.
08

Design principles

  • Match the delivery contract to the value of the data. Lossy where freshness wins (slots), guaranteed where completeness wins (tx status).
  • Keep the hot path free of network calls. Everything a submission needs is pre-fetched; the only in-window call is the send.
  • Submit ahead of the leader, not at it. Buffer the bundle before the slot begins.
  • One choke point for the block engine. Authentication and rate limiting live in a single function, live submissions prioritised over background traffic.
  • Classify from evidence, and admit ambiguity. Weigh competing causes; do not return false confidence.
  • Let the model reason, but bound it. A shadow baseline, a guardrail clamp, and a deterministic fallback keep the AI decision safe and auditable.
  • Measure, do not guess. Every hard problem in this build was solved by capturing a measurement that distinguished the candidate causes.
